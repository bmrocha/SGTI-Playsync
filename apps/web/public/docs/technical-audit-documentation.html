5. Autenticacao

Fluxo de autenticacao e mecanismos de seguranca

Fluxo de Autenticacao

sequenceDiagram participant U as Usuario participant F as Frontend participant A as API participant DB as Database U->>F: Email + Senha F->>A: POST /api/auth/login A->>DB: Validar credenciais DB-->>A: User data A->>A: Gerar JWT (RS256) A->>DB: Salvar refresh token A-->>F: Access + Refresh tokens F-->>U: Login sucesso Note over A,DB: Access token: 15min
Refresh token: 7 dias
Rotacao a cada uso

Mecanismos de Seguranca

  • JWT com RS256: Assinatura assimetrica com par de chaves publica/privada
  • Refresh Token Rotation: Novo refresh token gerado a cada renovacao
  • Senha Hash: bcrypt com salt round 12
  • 2FA (Opcional): Time-based One-Time Password (TOTP)
  • Session Management: Invalidacao por IP e user-agent
  • Rate Limiting: 100 requisicoes/minuto por IP

Duracao dos Tokens

Tipo Duracao Storage Risk
Access Token 15 minutos Memory (cliente) Baixo
Refresh Token 7 dias HttpOnly cookie Medio

7. Audit Logging

Eventos auditados e protecao anti-tampering

Visao Geral

O sistema implementa audit logging comprehensivo para todas as acoes criticas, seguindo principios de integridade e nao-repudiacao.

Eventos Auditados

  • Login/logout (sucesso e falha)
  • Alteracoes de usuarios (CRUD)
  • Upload e delecao de midias
  • Alteracoes de configuracoes
  • Modificacoes de permissoes
  • Exportacoes de dados
  • Erros de autenticacao

Estrutura do Log

interface AuditLog {
  id: string;              // UUID
  userId: string;          // Quem executou a acao
  action: string;          // Tipo de acao (CREATE, UPDATE, DELETE)
  resourceType: string;    // Tipo de recurso (User, Media, Playlist)
  resourceId: string;      // ID do recurso afetado
  metadata: JSON;          // Detalhes adicionais (IP, user-agent, antes/depois)
  timestamp: Date;         // Quando ocorreu
}

Protecao Anti-Tampering

  • Logs sao imutaveis (somente INSERT, nunca UPDATE/DELETE via API)
  • Cleanup automatico apos 180 dias via cron job
  • Acesso restrito ao perfil Admin
  • Backup regular do banco de audit logs

8. Backup e Disaster Recovery

Estrategia de backup e procedimento de recovery

Estrategia de Backup

Tipo Frequencia Retention Storage
Full Backup Diario (02:00 AM) 30 dias S3-compatible
Incremental A cada 6 horas 7 dias S3-compatible
WAL Archive Continuous 7 dias Local + S3

Objetivos de Recovery

  • RPO (Recovery Point Objective): 1 hora
  • RTO (Recovery Time Objective): 4 horas
  • Availability Target: 99.9% uptime

Procedimento de Recovery

  1. Identificar ponto de recovery no WAL archive
  2. Restaurar ultimo full backup
  3. Aplicar backups incrementais
  4. Replay WAL logs ate ponto desejado
  5. Validar integridade dos dados
  6. Redirecionar trafego para instancia recuperada

9. Conformidade LGPD

Principios implementados e direitos dos titulares

Principios Implementados

  • Finalidade: Coleta de dados apenas para fins especificos e legitimos
  • Transparencia: Politicas de privacidade claras e acessiveis
  • Seguranca: Criptografia, controle de acesso, audit logging
  • Necessidade: Minimo de dados necessarios coletados

Direitos dos Titulares

Direito Implementacao Prazo
Acesso aos dados Endpoint /api/privacy/export 15 dias
Retificacao Edicao via dashboard ou request 15 dias
Delecao (Right to be Forgotten) Endpoint /api/privacy/delete 15 dias
Portabilidade Export CSV/JSON via dashboard 15 dias
Revogacao de consentimento Configuracoes de privacidade Imediato

Medidas Tecnicas

  • Pseudonimizacao de dados pessoais em logs
  • Criptografia AES-256 para dados em repouso
  • TLS 1.3 para dados em transito
  • Controle de acesso granular (RBAC)
  • Audit logging de todas as operacoes com dados pessoais

12. Procedimentos de Pentest

Escopo do teste e ferramentas recomendadas

Escopo do Teste

  • Autenticacao e autorizacao
  • API endpoints (CRUD operations)
  • File upload/download
  • Session management
  • Rate limiting
  • Input validation

Contas de Teste

As contas de teste devem ser criadas exclusivamente para o ambiente de staging:

  • admin@test.playsync.com (Admin)
  • manager@test.playsync.com (Manager)
  • operator@test.playsync.com (Operator)
  • viewer@test.playsync.com (Viewer)

Ferramentas Recomendadas

  • OWASP ZAP: Scanner de vulnerabilidades web
  • Burp Suite: Interceptacao e teste de API
  • SQLMap: Teste de SQL injection
  • Nmap: Scanner de portas e servicos

Relatorio

Ao final do pentest, documentar:

  1. Vulnerabilidades encontradas (com severidade)
  2. Passos para reproducao
  3. Impacto potencial
  4. Recomendacoes de correcao
  5. Evidencias (screenshots, logs)
