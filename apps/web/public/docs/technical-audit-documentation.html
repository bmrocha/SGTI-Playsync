Arquitetura Geral do Sistema

Visao Geral

O SGTI PlaySync e uma plataforma de Digital Signage (Comunicacao Visual Corporativa) desenvolvida como monorepo, combinando frontend e backend em uma unica aplicacao Next.js 16 com App Router. O sistema permite gerenciamento centralizado de midias, criacao de playlists, agendamento inteligente e distribuicao para players em TVs corporativas.

Matriz de Controle de Acesso (RBAC)

Modelo de Autorizacao

O sistema implementa Role-Based Access Control (RBAC) com quatro niveis hierarquicos de permissao. Cada role determina acesso a rotas, funcionalidades e dados especificos. O isolamento por empresa (company_id) garante que usuarios de uma empresa nao acessem dados de outra.

Roles e Permissoes

Role Dashboard Midias Playlists Players Usuarios Empresas Auditoria Configuracoes
ADMIN [FULL] [CRUD] [CRUD] [CRUD] [CRUD] [CRUD] [READ] [FULL]
MANAGER [READ] [CRUD] [CRUD] [READ] [NONE] [NONE] [NONE] [READ]
EDITOR [READ] [CRUD] [CRUD] [READ] [NONE] [NONE] [NONE] [NONE]
VIEWER [READ] [READ] [READ] [READ] [NONE] [NONE] [NONE] [NONE]

Isolamento Multi-Empresa

Garantia de Isolamento: Todas as queries incluem clausula WHERE company_id = $X. Middleware de autenticacao injeta company_id do token JWT automaticamente, impedindo acesso cruzado entre empresas.

Verificacao de Permissoes

Fluxo de Autorizacao
  1. Autenticacao: Usuario fornece email + senha (+ 2FA se ativo)
  2. Token Generation: JWT contem userId, role, companyId
  3. Middleware: Todas as rotas protegidas verificam token via verifyToken()
  4. Role Check: API routes verificam role minimo necessario
  5. Company Filter: Queries filtram por company_id automaticamente

Mecanismos de Autenticacao e Sessao

Stack de Autenticacao

Implementacao baseada em JWT (JSON Web Tokens) utilizando biblioteca jose com algoritmo HS256 (HMAC-SHA256). Sistema de refresh tokens para renovacao automatica de sessoes sem requisitar login novamente.

Token Lifecycle

sequenceDiagram participant User as Usuario participant App as Next.js App participant DB as PostgreSQL User->>App: POST /api/auth/login App->>DB: Validar email + senha (bcrypt) DB-->>App: Hash valido App->>App: Gerar Access Token (1h) App->>App: Gerar Refresh Token (7d) App->>DB: Salvar refresh token em sessions App-->>User: { accessToken, refreshToken } Note over User,DB: Apos 1 hora User->>App: Request com accessToken expirado App-->>User: 401 Unauthorized User->>App: POST /api/auth/refresh App->>DB: Validar refreshToken DB-->>App: Token valido App->>App: Gerar novo accessToken App-->>User: { accessToken } Note over User,DB: Logout User->>App: POST /api/auth/logout App->>DB: Deletar refresh token DB-->>App: Sucesso App-->>User: 200 OK

Detalhes dos Tokens

Token Duracao Algoritmo Payload Armazenamento
Access Token 1 hora (3600s) HS256 userId, email, role, companyId, tokenType Memory (Zustand)
Refresh Token 7 dias HS256 userId, email, tokenType DB (sessions table)

Two-Factor Authentication (2FA)

Implementacao TOTP
  • Biblioteca: otplib v13.3.0 com NobleCryptoPlugin e ScureBase32Plugin
  • Algoritmo: TOTP (Time-based One-Time Password) RFC 6238
  • Compatibilidade: Google Authenticator, Authy, Microsoft Authenticator
  • Setup: QR Code gerado via qrcode library com otpauth URL
  • Temp Secret: Expira em 10 minutos para seguranca
  • Verificacao: Stateless durante login, stateful durante setup

Seguranca de Senhas

Mecanismo Implementacao Configuracao
Hash bcryptjs Salt rounds: 10
First Login Forca troca de senha Flag force_password_change
Lockout Progressivo por tentativas Lock apos 5 falhas
Reset Via db:reset-admin Usa ADMIN_INITIAL_PASSWORD

Rate Limiting

Protecao contra Brute Force: Sistema de rate limiting implementado via migration 005_add_rate_limits_and_refresh.sql. Bloqueio progressivo de login apos falhas consecutivas, com campos failed_attempts e locked_until na tabela users.

Registro de Logs e Auditoria

Sistema de Logging

O PlaySync implementa logging em tres niveis: application logs (Pino), audit logs (database) e error tracking (Sentry). Cada camada serve a um proposito especifico de observabilidade e compliance.

Tipos de Logs

Tipo Tecnologia Armazenamento Exemplo
Application Logs Pino (structured JSON) Docker logs (stdout/stderr) API requests, errors, warnings
Audit Logs PostgreSQL (audit_logs table) Banco de dados (persistente) CRUD operations, login events
Playback Logs PostgreSQL (playback_logs table) Banco de dados (persistente) Media exibida, player, timestamp
Error Tracking Sentry APM Sentry Cloud (externo) Stack traces, user context, breadcrumbs

Estrutura do Audit Log

CREATE TABLE audit_logs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    action VARCHAR(50) NOT NULL,          -- 'CREATE', 'UPDATE', 'DELETE', 'LOGIN'
    entity_type VARCHAR(50) NOT NULL,     -- 'playlist', 'media', 'user', 'player'
    entity_id UUID NOT NULL,
    changes JSONB,                        -- Before/after state
    ip_address VARCHAR(45),
    created_at TIMESTAMP DEFAULT NOW()
);

Politica de Retencao

Tipo de Log Retencao Justificativa
Audit Logs 365 dias (1 ano) Conformidade LGPD + Auditoria
Playback Logs 180 dias (6 meses) Analytics + Troubleshooting
Application Logs 30 dias Docker rotation (10MB x 3 files)
Sentry Events 90 dias Plano Sentry (configuravel)

Integridade Anti-Tampering

Protecoes Implementadas:
  • Audit logs sao append-only - nao ha endpoints para DELETE ou UPDATE
  • Timestamps gerados pelo servidor (NOW()), nao pelo cliente
  • IP address capturado automaticamente do request
  • Changes em formato JSONB com diff before/after
  • Acesso restrito a role admin para leitura

Anonimizacao e Pseudonimizacao

Medias de Protecao:
  • Senhas: Nunca armazenadas em texto puro - hash bcrypt com salt
  • IP Addresses: Registrados em logs de auditoria (base legal: legitimo interesse)
  • Tokens JWT: Pseudonimizados (contem userId, nao dados pessoais)
  • Dados de uso: Playback logs anonimos (sem identificacao de usuario)

Transferencia Internacional

Sentry APM: Dados de erro podem ser transmitidos para servidores Sentry (potencialmente fora do Brasil). Recomendado configurar NEXT_PUBLIC_SENTRY_DSN com instancia self-hosted se necessario compliance estrito.

Protecao contra Ameacas

SQL Injection

Status: PROTEGIDO - Todas as queries utilizam parametros ($1, $2, ...) via driver pg. Nenhum concatenacao de strings em queries SQL. Validacao adicional via zod schemas.

Cross-Site Scripting (XSS)

Mitigacoes
  • React 19: Auto-escaping de outputs por default
  • CSP: script-src 'self' 'unsafe-inline' (restrito a origins conhecidos)
  • Sanitizacao: Inputs validados via Zod schemas
  • Next.js: Protecao built-in contra XSS em App Router

Cross-Site Request Forgery (CSRF)

Status: PROTEGIDO - Next.js App Router utiliza Server Actions com protecao CSRF automatica. API routes autenticadas via JWT (Bearer tokens) - CSRF nao se aplica a APIs stateless.

Insecure Direct Object Reference (IDOR)

Protecao Implementada
  1. Todas as queries incluem WHERE company_id = $X
  2. Middleware injeta companyId do token JWT
  3. UUIDs nao sao sequenciais (dificulta enumeration)
  4. Role checks previnem acesso vertical

Rate Limiting

Endpoint Limite Acao ao Exceder
POST /api/auth/login 5 tentativas / 15 minutos Lockout progressivo + locked_until
POST /api/auth/2fa/verify 10 tentativas / 10 minutos Bloqueio temporario do endpoint
API routes gerais 100 requests / minuto HTTP 429 Too Many Requests

Outras Ameacas

Ameaca Protecao Status
Brute Force bcrypt salt 10 + lockout progressivo PROTEGIDO
Session Hijacking JWT HS256 + refresh tokens em DB PROTEGIDO
Man-in-the-Middle HSTS + TLS 1.3 recomendado PROTEGIDO
Clickjacking X-Frame-Options: SAMEORIGIN PROTEGIDO
MIME Sniffing X-Content-Type-Options: nosniff PROTEGIDO
File Upload Attack Validacao de tipo + tamanho + path sanitization PROTEGIDO
