1 Requisitos do Sistema

💻 Servidor
  • Linux (Ubuntu 22.04+ / Debian 12+) ou Windows Server 2019+
  • Node.js ≥ 22 LTS
  • npm (v10+) — incluso no Node.js
  • PostgreSQL 15 ou superior
  • Docker & Docker Compose (opcional)
  • Git
📶 Rede
  • Porta 3000 liberada (Next.js)
  • Porta 5432 (PostgreSQL — interno)
  • Acesso à internet para dependências npm
  • DNS configurado (se usar domínio próprio)
📀 Armazenamento
  • Mínimo 2 GB de RAM
  • 10 GB de disco para sistema + uploads
  • Swap configurado (recomendado 2 GB)
⚙️ TV Agent (players)
  • Linux / Raspberry Pi (ARM) / Windows
  • Python 3.8+
  • VLC (player de mídia)
  • Conexão de rede com o servidor

2 Preparação do Ambiente

ℹ️ Node.js via nvm (recomendado)

Instale o Node.js 22 através do nvm para gerenciar versões facilmente:

# Instalar nvm
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
source ~/.bashrc

# Instalar Node.js 22 LTS
nvm install 22
nvm use 22
nvm alias default 22

# Verificar instalação
node --version   # v22.x.x
npm --version    # 10.x.x
📦 Download do Projeto
# Clonar o repositório
git clone https://github.com/seu-org/SGTI-Playsync.git
cd SGTI-Playsync

# Ou extrair o arquivo ZIP no diretório desejado
unzip SGTI-Playsync.zip -d /opt/sgti-playsync
cd /opt/sgti-playsync

3 Configuração do Banco de Dados

3.1 PostgreSQL — Instalação

# Ubuntu/Debian
sudo apt update
sudo apt install postgresql postgresql-contrib -y
sudo systemctl enable postgresql
sudo systemctl start postgresql

# Verificar versão
psql --version

3.2 Criar Database e Usuário

# Acessar o PostgreSQL como superusuário
sudo -u postgres psql

postgres=# CREATE USER sgti_admin WITH PASSWORD 'sua_senha_forte_aqui';
postgres=# CREATE DATABASE sgti_playsync OWNER sgti_admin;
postgres=# GRANT ALL PRIVILEGES ON DATABASE sgti_playsync TO sgti_admin;
postgres=# \c sgti_playsync
sgti_playsync=# GRANT ALL ON SCHEMA public TO sgti_admin;
sgti_playsync=# \q
⚠️ Senha Forte

Use uma senha de no mínimo 24 caracteres com letras maiúsculas, minúsculas, números e caracteres especiais. A string de conexão será usada em vários lugares.

3.3 Conectar ao Banco

A string de conexão segue o formato:

DATABASE_URL="postgresql://sgti_admin:sua_senha_forte_aqui@localhost:5432/sgti_playsync?schema=public"

3.4 Migrations

Com o Node.js já instalado, execute as migrations para criar as tabelas:

# Instalar dependências do banco
cd /opt/sgti-playsync
npm install

# Exportar variável de ambiente temporária
export DATABASE_URL="postgresql://sgti_admin:sua_senha_forte_aqui@localhost:5432/sgti_playsync"

# Executar migrations
npm run db:migrate

# Verificar se tudo está correto
npm run db:verify

3.5 Seed (criar admin inicial)

# Povoar dados iniciais (cria usuário admin padrão)
npm run db:seed

# Opcional: redefinir senha do admin (se necessário)
npm run db:reset-admin
✅ Credenciais Padrão

Após o seed, o sistema cria um admin com as credenciais definidas no script de seed. Verifique o arquivo packages/database/src/db/seed.ts para ver o email e senha padrão.

3.6 Apenas Docker? (pule esta etapa)

Se for usar Docker Compose, o banco é criado automaticamente — vá direto para a seção 6.

4 Variáveis de Ambiente

Crie o arquivo .env na raiz do projeto (copie de .env.example se existir):

# ============================================
# OBRIGATÓRIAS
# ============================================

# String de conexão PostgreSQL
DATABASE_URL="postgresql://sgti_admin:sua_senha_forte_aqui@localhost:5432/sgti_playsync?schema=public"

# Chave secreta para JWT (HS256) — gere com: openssl rand -hex 64
JWT_SECRET="sua_chave_jwt_aqui_com_pelo_menos_64_caracteres_hex"

# URL pública da aplicação
NEXT_PUBLIC_API_URL="http://10.1.191.107:3000"

# ============================================
# LICENCIAMENTO (obrigatório em produção)
# ============================================

# Emissor da licença (padrão: SGTI)
LICENSE_ISSUER="SGTI"

# Chave pública RSA em formato PEM (ou base64)
LICENSE_PUBLIC_KEY="-----BEGIN PUBLIC KEY-----\nMIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQEA...\n-----END PUBLIC KEY-----"

# ID de instalação (opcional — gerado automaticamente se omitido)
INSTALLATION_ID="meu-servidor-producao-01"

# ============================================
# OPCIONAIS
# ============================================

# SSL do banco de dados (true/false)
DATABASE_SSL=false
# Caminho para certificado CA (se SSL=true)
# DATABASE_SSL_CA_PATH=/etc/ssl/certs/ca-certificates.crt

# Nível de log: debug | info | warn | error
LOG_LEVEL="info"

# Emails que ignoram 2FA (separados por vírgula)
BYPASS_2FA_EMAILS="admin@exemplo.com"

# Chave secreta para chamadas cron (deve bater com header Authorization)
CRON_SECRET="sua_chave_cron_aqui"

# Sentry DSN (opcional — para rastreamento de erros)
# NEXT_PUBLIC_SENTRY_DSN=https://xxx@xxx.ingest.sentry.io/xxx
# SENTRY_DSN=https://xxx@xxx.ingest.sentry.io/xxx

# Timezone do container Docker
TZ="America/Sao_Paulo"

4.1 Gerar JWT_SECRET

# Gere uma chave segura com OpenSSL
openssl rand -hex 64

# Saída esperada (exemplo):
7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d3e4f5a6b7c8d9e0f1a2b3c4d5e6f7a8b

4.2 Tabela completa de variáveis

Variável Obrigatória Descrição
DATABASE_URL Sim String de conexão PostgreSQL
JWT_SECRET Sim Chave HMAC-SHA256 para JWT. Mínimo 64 caracteres hex.
NEXT_PUBLIC_API_URL Sim URL base da aplicação (ex: http://192.168.1.100:3000)
LICENSE_ISSUER Não (padrão: SGTI) Emissor da licença
LICENSE_PUBLIC_KEY Sim (produção) Chave pública RSA em formato PEM para validação de licença
INSTALLATION_ID Não Identificador único da instalação
DATABASE_SSL Não true para habilitar SSL na conexão com o banco
DATABASE_SSL_CA_PATH Não Caminho para certificado CA (necessário se SSL=true)
LOG_LEVEL Não debug | info | warn | error
BYPASS_2FA_EMAILS Não Lista de emails separados por vírgula que pulam 2FA
CRON_SECRET Não Token para autenticar chamadas aos endpoints cron
NEXT_PUBLIC_SENTRY_DSN Não Sentry DSN (cliente)
SENTRY_DSN Não Sentry DSN (servidor)
NODE_ENV Não development | production
NEXT_TELEMETRY_DISABLED Não 1 para desabilitar telemetria Next.js

5 Instalação Manual

🏠 Produção (bare metal)

Execute os comandos abaixo no servidor para fazer deploy manual.

# 1. Instalar dependências do monorepo
cd /opt/sgti-playsync
npm install

# 2. Exportar variáveis de ambiente (ou usar .env)
export $(cat .env | xargs)

# 3. Executar migrations no banco
npm run db:migrate
npm run db:seed

# 4. Fazer build de produção
npm run build

# 5. Iniciar o servidor
npm start
✅ Aplicação no ar

Acesse http://SEU_IP:3000 no navegador.

5.1 Process Manager (PM2) — Recomendado

Para manter o servidor rodando em segundo plano e reiniciar automaticamente após falhas:

# Instalar PM2 globalmente
npm install -g pm2

# Iniciar com PM2
pm2 start npm --name "sgti-playsync" -- start
pm2 save
pm2 startup   # Configurar restart automático na inicialização do sistema

# Comandos úteis
pm2 status
pm2 logs sgti-playsync
pm2 restart sgti-playsync
pm2 stop sgti-playsync

5.2 Proxy Reverso (Nginx) — Opcional

# /etc/nginx/sites-available/sgti-playsync
server {
    listen 80;
    server_name sgti.exemplo.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}
sudo ln -s /etc/nginx/sites-available/sgti-playsync /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx

6 Implantação com Docker

🐳 Mais simples e isolado

O Docker Compose sobe a aplicação e o PostgreSQL em containers próprios.

6.1 Pré-requisitos

# Instalar Docker
curl -fsSL https://get.docker.com | sudo bash
sudo usermod -aG docker $USER

# Instalar Docker Compose (plugin)
sudo apt install docker-compose-plugin -y

# Verificar
docker --version
docker compose version

6.2 Configurar .env

Edite o arquivo .env na raiz do projeto. Para Docker, use o nome do serviço como host do banco:

# .env — para Docker
DATABASE_URL="postgresql://sgti_admin:sua_senha_forte_aqui@playsync_db_player:5432/sgti_playsync?schema=public"
JWT_SECRET="sua_chave_hex_aqui"
NEXT_PUBLIC_API_URL="http://SEU_IP:3000"
POSTGRES_USER="sgti_admin"
POSTGRES_PASSWORD="sua_senha_forte_aqui"
POSTGRES_DB="sgti_playsync"
TZ="America/Sao_Paulo"
⚠️ Host do banco

No Docker, o host do PostgreSQL é o nome do serviço: playsync_db_player, não localhost.

6.3 Build & Executar

# Construir e iniciar todos os serviços
docker compose up -d --build

# Acompanhar logs
docker compose logs -f

# Verificar status
docker compose ps

# Parar tudo
docker compose down

# Parar e remover volumes (dados do banco serão perdidos!)
docker compose down -v

6.4 Verificar se está funcionando

# Health check do container
docker compose ps

# Testar endpoint de saúde
curl http://localhost:3000/api/health

# Testar setup do banco via API
curl http://localhost:3000/api/setup-db

6.5 Volumes importantes

VolumeContainerDescrição
postgres_data PostgreSQL Dados persistentes do banco
./public/uploads App Uploads de mídia dos usuários
./.env App Arquivo de configuração montado como volume

6.6 Acesso ao banco via host

O banco de dados fica exposto na porta 127.0.0.1:5433 do host (não 5432). Conecte-se com:

psql -h 127.0.0.1 -p 5433 -U sgti_admin -d sgti_playsync

7 Primeiro Acesso & Pós-implantação

7.1 Login

  1. Acesse http://SEU_IP:3000
  2. Faça login com o email e senha criados pelo seed
  3. Se for o primeiro acesso, o sistema pode solicitar troca de senha e configuração de 2FA
⚠️ Licenciamento

Em produção, o sistema exige uma licença válida. Acesse Configurações → Licenciamento para ativar. Em ambiente de desenvolvimento (NODE_ENV=development), a licença não é exigida.

7.2 Checklist pós-implantação

  • ✅ Definir senha forte para o admin
  • ✅ Configurar 2FA (recomendado para todos os usuários)
  • ✅ Ativar licença do sistema
  • ✅ Revisar configurações de segurança (IPs confiáveis, lockout)
  • ✅ Configurar integração de weather (opcional)
  • ✅ Testar upload de mídia
  • ✅ Criar pelo menos uma empresa e uma playlist
  • ✅ Configurar cron de limpeza de logs (veja seção 9)

7.3 Criar primeiro player

  1. Vá para Players no menu lateral
  2. Clique em Novo Player
  3. Preencha nome, empresa e clique em salvar
  4. Copie o token gerado — será usado no TV Agent

8 Configuração do TV Agent

📺 Player nas TVs

O TV Agent é um script Python que roda no dispositivo (Raspberry Pi, Linux, Windows) e exibe o conteúdo do PlaySync.

8.1 Requisitos no dispositivo

  • Python 3.8+
  • VLC instalado
  • Conexão de rede com o servidor PlaySync

8.2 Instalação via script

Existe um script automatizado no projeto:

# Copiar o script para o dispositivo ou executar diretamente
cd /opt/sgti-playsync/apps/web
chmod +x install_simple_tvagent.sh
./install_simple_tvagent.sh

8.3 Configuração manual

# Instalar dependências Python
pip install requests python-vlc

# Criar arquivo de configuração /opt/playsync-agent/config.json
{
    "server_url": "http://SEU_IP:3000",
    "player_id": "ID_DO_PLAYER",
    "token": "TOKEN_DO_PLAYER",
    "refresh_interval": 30
}

# Executar o agente
python3 /opt/playsync-agent/agent.py
⚠️ Firewall

Certifique-se de que o dispositivo TV Agent consegue acessar a porta 3000 do servidor. Teste com: curl http://SEU_IP:3000/api/health

9.2 Cron de Limpeza de Logs

Configure uma tarefa cron no servidor para limpar logs de auditoria antigos:

# /etc/cron.d/sgti-cleanup
0 2 * * * root curl -X POST http://localhost:3000/api/cron/cleanup-logs \
  -H "Authorization: Bearer SUA_CRON_SECRET"

# Ou no crontab do usuário
crontab -e
0 2 * * * curl -X POST http://localhost:3000/api/cron/cleanup-logs \
  -H "Authorization: Bearer SUA_CRON_SECRET"

9.4 Problemas comuns

💥 Erro: "JWT_SECRET não configurado"

O arquivo .env não está sendo carregado. Verifique se ele existe na raiz e se as variáveis estão corretamente definidas. No Docker, certifique-se de que .env está montado como volume.

💥 Erro: "ECONNREFUSED" no banco de dados

Verifique se o PostgreSQL está rodando e se a DATABASE_URL está correta. No Docker, aguarde o banco ficar saudável antes da aplicação.

💥 Erro: "License not activated"

O sistema exige licença em produção. Configure LICENSE_PUBLIC_KEY e aplique a licença em Configurações → Licenciamento. Para desenvolvimento, use NODE_ENV=development.

💥 Erro: "Refresh token inválido"

A sessão expirou (refresh token tem validade de 7 dias). Faça login novamente. Se o erro persistir, verifique se o campo refresh_token existe na tabela sessions (rode npm run db:migrate).

💥 Build falha com erro de tipo TypeScript

Execute npm run typecheck para identificar os erros. Verifique se todas as dependências foram instaladas com npm install.

9.5 Backup do banco de dados

# Backup
pg_dump -U sgti_admin -h localhost sgti_playsync > backup_$(date +%Y%m%d).sql

# Restore
psql -U sgti_admin -h localhost sgti_playsync < backup_20260528.sql

# Docker
docker exec -t playsync_db_player pg_dump -U sgti_admin sgti_playsync > backup.sql

NívelQuando usar

infoProdução — eventos normais do sistema
warnAtenção — algo inesperado, mas não crítico
errorErro — requer investigação

Logs de auditoria são armazenados no banco (tabela audit_logs) e podem ser visualizados no dashboard em Auditoria.
